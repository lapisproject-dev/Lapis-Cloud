== Context

This document describes the data model introduced with V0.1.5: contribution management,
document storage, and communication (mailing lists + direct messages). `Member` and `Account` are
modelled here only as a **Foundation stub** — as granular as the three V0.1.5 areas need them as
foreign keys. The full member master-data model (application/withdrawal workflow, role and
permission concept, auth/session) follows in V0.1.2-V0.1.4 and replaces this stub without breaking
the foreign keys defined here.

Persistence generation: the `uml-to-erm`/`erm-to-exposed` MDA pipeline (ADR-0016, kUML modules
`kuml-transform-uml-to-erm` + `kuml-codegen-m2m-exposed`, Maven Central) has existed since V3.4.x —
the `dev.kuml` Gradle *plugin* itself, however, is unpublished. For **Foundation**
(`Member`/`Account`, this stub; `lapis-server/src/main/kuml/00-foundation.kuml.kts`),
**contribution management** (`MembershipTier`/`Contribution`;
`lapis-server/src/main/kuml/01-contribution.kuml.kts`), **document storage**
(`DocumentFolder`/`Document`/`DocumentVersion`;
`lapis-server/src/main/kuml/02-document.kuml.kts`), and **communication**
(`MailingList`/`MailingListSubscription`/`MailingMessage`/`MailingDeliveryLog`/`DirectMessage`;
`lapis-server/src/main/kuml/03-communication.kuml.kts`), these `.kuml.kts` files have, since the
MDA persistence retrofit wave, been the versioned source-of-truth model — verified per domain by a
dedicated `SchemaDriftTest`
(`lapis-server/src/test/kotlin/network/lapis/cloud/server/db/{SchemaDriftTest,
ContributionSchemaDriftTest, DocumentSchemaDriftTest, CommunicationSchemaDriftTest}.kt`) against
the real Flyway-migrated schema and the hand-written Exposed `Table` objects — no longer purely
documentary. The hand-written Exposed tables under
`lapis-server/src/main/kotlin/.../db/tables` remain, deliberately, the compiled/actually-imported
artifact (designModelStrategy option B: the enum-to-VARCHAR downgrade and the missing Kotlin
object-name override currently make a full generated-and-compiled swap lossy/disruptive for this
repository — see the header comments in the respective `.kuml.kts` files and CLAUDE.md's
kUML-Repo-Konventionen in the vault for the full rationale). The document-storage domain brought
one further modelling trade-off beyond Foundation/contribution management: `document.folder_id`,
`document.created_by`, and `document_version.uploaded_by` are modelled as plain `«Column»` UUID
attributes rather than UML associations, because `UmlToErmTransformer`'s association-to-FK
column-name derivation (`snake_case(singular(targetClass)) + "_id"`) doesn't match the real column
names and no DSL-side override exists (details in `02-document.kuml.kts`'s header comment). The
communication domain confirms the same naming-derivation trade-off (`mailing_list.created_by`,
`mailing_message.sent_by`) and additionally surfaces the case the retrofit plan flagged as a risk —
two associations from the same class to the same target: `direct_message.sender_id` and
`direct_message.recipient_id` both reference `member`, but `UmlToErmTransformer`'s role-name-based
collision resolution only ever yields `member_id`/`<role>_id` for a genuine association pair, never
both desired column names together — verified empirically during this wave. Both foreign keys are
therefore likewise modelled as plain `«Column»` attributes (details in `03-communication.kuml.kts`'s
header comment and `CommunicationSchemaDriftTest`). With this, all four V0.1.5 domains (Foundation,
contribution management, document storage, communication) are no longer purely documentary but each
verified by its own `SchemaDriftTest`.

== Accounting (SKR42 + double-entry, V0.3.1, chart swapped from SKR49 in V0.3.1.1)

`LedgerAccount`/`JournalEntry`/`Posting` (`lapis-server/src/main/kuml/10-accounting.kuml.kts`,
verified by `AccountingSchemaDriftTest`) implement double-entry bookkeeping on a representative
SKR42 chart of accounts (DATEV's current Kontenrahmen for Vereine, Stiftungen und gGmbHs, based on
SKR04; it replaced SKR49, which DATEV has maintained no further since 01.01.2025). `LedgerAccount`
is deliberately not named `Account` — that name is already taken by the Foundation domain's
user-login `Account`/`AccountRole`.

A SKR42 account number's leading digit (`LedgerAccount.accountClass`, 0-9) is the Kontenklasse.
Unlike the retired SKR49, SKR42 does *not* pre-partition the four Gemeinnützigkeit spheres into
account-number ranges, and the Kontenklassen do not separate income from expense the way a naive
reading might suggest: class 4 holds *all* Erträge/Umsatzerlöse across all four spheres (e.g. 4000
Mitgliedsbeiträge, 4045 Spenden, 4200/4201 Erlöse/Eintrittsgelder), class 5 is itself an *expense*
class (Wareneingang / Aufwendungen für Roh-, Hilfs- und Betriebsstoffe, e.g. 5000/5200
Wareneingang), class 6 covers the remaining operating expenses (e.g. 6000 Löhne und Gehälter, 6310
Miete), and class 7 is the Finanzergebnis (e.g. 7110 Zinserträge, 7300 Zinsaufwand) — covering both
income and expense sub-accounts. None of classes 4-7 are sphere-partitioned; the same account can
be booked to any of the four spheres. Sphere is assigned per booking via a cost center (DATEV
KOST1), not derived from the account. `accountClass` remains a reporting/grouping field only; the
real per-posting cost-center/sphere attribute is `Posting.sphere` -- see
<<four-sphere-separation,Four-sphere Gemeinnützigkeit separation (V0.3.3)>> below.

`JournalEntry.status` is `DRAFT` (incomplete/unbalanced postings allowed, freely editable) or
`POSTED` (immutable, validated balanced at the moment of transition). The balance invariant
(Σdebit = Σcredit per `JournalEntry`) is a cross-row aggregate that no CHECK constraint or index
can express — it is enforced exclusively at the service layer
(`network.lapis.cloud.server.rpc.AccountingService`, backed by the pure
`network.lapis.cloud.server.rpc.JournalEntryBalance` helper) inside the same transaction that
writes the entry and its postings, so an unbalanced attempt never leaves a partial write behind.
`network.lapis.cloud.server.rpc.GeneralLedgerCalculator` computes the Hauptbuch (general ledger)
running balance per `LedgerAccount`, signed by that account's `LedgerAccountType` normal-balance
side (`ASSET`/`EXPENSE` debit-normal, `LIABILITY`/`EQUITY`/`INCOME` credit-normal).

=== Financial statements (V0.3.2)

`network.lapis.cloud.server.rpc.FinancialStatementCalculator` derives three read-only reports
purely from the existing `POSTED` postings — no new persistence, no schema change, and therefore
no new `*SchemaDriftTest`. `IncomeStatementDto`/`BalanceSheetDto`/`AnnualFinancialStatementDto`/
`StatementLineDto` (`lapis-shared/.../domain/FinancialStatements.kt`) are non-persisted reporting
DTOs, exactly like the existing `GeneralLedgerDto`/`GeneralLedgerLineDto`. Three new read-only
`IAccountingService`/`AccountingService` methods expose them, gated by the same TREASURER/BOARD/
ADMIN roles as `getGeneralLedgerAccount`.

* **Gewinn- und Verlustrechnung (GuV / `getIncomeStatement`)** — the flow of `INCOME`/`EXPENSE`
  postings over `[from, to]` (`from == null` means "since inception"). `result` = `totalIncome` −
  `totalExpense`; negative is a Jahresfehlbetrag.
* **Bilanz (`getBalanceSheet`)** — the *cumulative* stock of postings since inception through a
  single `asOf` cutoff, deliberately **not** windowed to a fiscal year. This is the load-bearing
  design decision: `GeneralLedgerDto.openingBalance` is always `0` (no Saldenvortrag/carryforward
  support yet — see that DTO's KDoc), so there is nothing to seed a fiscal-year-windowed Bilanz
  with. Because `INCOME`/`EXPENSE` are never closed to `EQUITY` (no closing entries generated in
  this wave), the Bilanz only balances (Aktiva == Passiva) if the accumulated result
  (Σ`INCOME` − Σ`EXPENSE` through `asOf`) is shown as its own derived equity line
  (`BalanceSheetDto.accumulatedResult`) — this is guaranteed to reconcile by the
  server-enforced Σdebit = Σcredit invariant, and `balanced` is asserted `true` as a regression
  guard in `FinancialStatementCalculatorTest`/`AccountingServiceTest`.
* **Jahresabschluss (`getAnnualFinancialStatement`)** — assumes Geschäftsjahr = Kalenderjahr
  (calendar year) and bundles the GuV of `[Jan 1, Dec 31]` of the given year with the Bilanz as of
  that year's Dec 31. It reports two honest, distinct figures rather than conflating them:
  `periodResult` (that year's GuV flow) and `accumulatedResult` (the inception-cumulative Bilanz
  equity result) — these coincide in the very first fiscal year and legitimately diverge from the
  second year on, since nothing is closed/carried forward between years.

Line lists (`StatementLineDto`) omit accounts with a zero net balance in the requested scope — they
are a derived-from-journal view, not a complete chart-of-accounts listing. Still explicitly out of
scope (unchanged from V0.3.1's list above): fiscal-year close/Saldenvortrag, Kassenbuch (V0.3.5),
Kostenstellen (V0.3.6), non-calendar fiscal years. (Mittelverwendungsrechnung/Rücklagen, originally
listed here as a later non-goal, shipped in V0.3.4 -- see below.)

=== Four-sphere Gemeinnützigkeit separation (V0.3.3)

`Posting.sphere` (`network.lapis.cloud.shared.domain.GemeinnuetzigkeitSphere`) is a mandatory
(`NOT NULL`) per-posting enum column added to `lapis-server/src/main/kuml/10-accounting.kuml.kts`,
tagging every booking line with exactly one of the four legally-mandated Gemeinnützigkeit spheres
(§§ 51-68 AO): `IDEELLER_BEREICH`, `VERMOEGENSVERWALTUNG`, `ZWECKBETRIEB`,
`WIRTSCHAFTLICHER_GESCHAEFTSBETRIEB` (DATEV KOST1 codes 1-4). This is the highest-legal-risk wave
in the accounting backlog: losing the strict separation between these spheres risks the
organization's tax-exempt status, so the guarantee that every posting names exactly one sphere is
enforced structurally at three layers rather than by a single service-layer check —

* the Kotlin type: `PostingInput.sphere`/`PostingDto.sphere` are non-nullable with **no default
  value**, so every call site must supply one explicitly;
* the RPC wire format: `kotlinx.serialization` enum decoding rejects any payload that omits or
  mis-names the field;
* the database: the generated `sphere` column is `NOT NULL` with a `CHECK` constraint restricting
  it to the four real literals.

**Placement**: per-`Posting` (booking line), not `JournalEntry` and not `LedgerAccount`. As the
SKR42 discussion above establishes, sphere is *not* derivable from the account/Kontenklasse — it is
assigned per booking via DATEV KOST1, which is inherently a per-line cost center. A per-line
attribute is also the only placement that lets one balanced `JournalEntry` legitimately record an
inter-sphere transfer (Umbuchung): a debit line in one sphere, a credit line in another. This
deliberately also forces a sphere onto Bestandskonten lines (Kasse/Bank), which are conceptually
sphere-neutral — accepted tradeoff, since the four-sphere report below only aggregates
`INCOME`/`EXPENSE`-typed accounts, so a cash line's sphere is informational only.

**No Sammelposten/KOST9 escape bucket**: `GemeinnuetzigkeitSphere` has exactly the four real
spheres, on purpose — a fifth "unassigned"/collective literal would be a way to *not* assign a
sphere, which contradicts strict separation. A general cost-center/Kostenstellen allocation
mechanism remains V0.3.6 scope.

**A single `JournalEntry` may span multiple spheres**: no single-sphere-per-entry invariant is
enforced, because sphere is per-line — a cross-sphere Umbuchung is fully transparent since each
line names its own sphere. Inter-sphere transfers are representable this way but are not specially
validated, tracked, or reported this wave; a dedicated §58 AO Mittelweitergabe workflow is deferred
to V0.3.4 (Mittelverwendungsrechnung).

**Reporting payoff** — `getFourSphereIncomeStatement` (`FourSphereIncomeStatementDto`,
`SphereResultDto`, both in `lapis-shared/.../domain/FinancialStatements.kt`; derivation in
`FinancialStatementCalculator.fourSphereIncomeStatement`): the same `INCOME`/`EXPENSE` flow over
`[from, to]` as `getIncomeStatement`, re-aggregated per sphere instead of collapsed across all
four. `spheres` always contains exactly four entries in `GemeinnuetzigkeitSphere` enum order,
zero-filled if a sphere had no activity — so a UI can always render a fixed four-row layout. The
overall `result` equals both the sum of the four per-sphere results and the plain
`getIncomeStatement` result for the identical window: sphere is purely an orthogonal
re-aggregation of the same postings, not a different scope. Gated by the same
TREASURER/BOARD/ADMIN roles as the other statement methods.

Explicitly out of scope for V0.3.3: a four-sphere balance sheet, §64 Abs.3 AO
Freigrenze/taxability evaluation of the wirtschaftlicher Geschäftsbetrieb surplus, and any
dedicated inter-sphere-transfer tracking workflow — all deferred to later waves.

=== Mittelverwendungsrechnung + Rücklagen (V0.3.4)

**This is a treasurer's Nachweis (evidence) aid for §55 AO ("zeitnahe Mittelverwendung") and §62 AO
(Rücklagenbildung) -- it is NOT an automated Gemeinnützigkeit compliance verdict.** It is the
second-highest legal-risk wave in the accounting backlog after V0.3.3, and is scoped deliberately
conservatively: no percentage-cap enforcement, no automatic exemption application, no certification
of tax-exempt status one way or the other.

**Reserves are ordinary `EQUITY` ledger accounts, not a new persisted entity.** A treasurer moves
funds into a reserve via a normal balanced double-entry posting (e.g. debit a general equity
account, credit the reserve account) -- exactly the same mechanism as every other booking in this
codebase. The *only* schema addition this wave makes is one nullable enum column,
`LedgerAccount.reserveType`
(`network.lapis.cloud.shared.domain.ReserveType`: `PROJEKTRUECKLAGE`, `FREIE_RUECKLAGE`,
`WIEDERBESCHAFFUNGSRUECKLAGE`, `BETRIEBSMITTELRUECKLAGE`), which makes an `EQUITY` account
machine-classifiable by its §62 AO reserve category. A dedicated `Reserve`/`ReserveAllocation`
entity was deliberately rejected as over-scoped -- it would duplicate the ledger that already
tracks these balances via ordinary postings. `reserveType` may only be set on an `EQUITY` account
(`AccountingService.requireReserveTypeOnlyOnEquity`, a service-layer cross-column guard, same class
of constraint as the balance invariant -- not expressible as a single-row `CHECK`). §62 Abs.1 Nr.4
(Rücklage zum Erwerb von Gesellschaftsrechten) is not offered as a literal -- out of scope, rare for
a Verein/Partei. **The §62 Abs.1 Nr.3 freie-Rücklage percentage cap is deliberately NOT enforced in
code** -- it is a number a human must configure and verify against current law (see
`ReserveType.FREIE_RUECKLAGE` KDoc); this codebase does not hard-code an assumed percentage.

**`network.lapis.cloud.server.rpc.UseOfFundsCalculator`** (pure, unit-tested without a database,
same idiom as `FinancialStatementCalculator`/`GeneralLedgerCalculator`/`JournalEntryBalance`)
derives the read-only `UseOfFundsStatementDto` (`lapis-shared/.../domain/UseOfFunds.kt`) from the
existing `POSTED` postings plus each account's `reserveType`:

* **Mittelzufluss/Mittelverwendung** (`UseOfFundsYearDto.fundsReceived`/`fundsUsed`) -- the year's
  `INCOME`/`EXPENSE` flow, same derivation as the GuV.
* **Rücklagen-Zuführung** (`fundsAllocatedToReserves`, `reserveMovements`) -- the year's net
  Zuführung to `EQUITY` accounts carrying each `ReserveType`, broken down per type (always four
  entries, zero-filled), plus each type's cumulative closing balance.
* **Per-sphere disaggregation** (`receivedBySphere`/`usedBySphere`) -- purely informational; always
  four `GemeinnuetzigkeitSphere` entries, zero-filled.
* **Mittelvortrag / §55 AO timely-use clock** (`timelyUseObligationRemaining`, `overdueAmount`) -- a
  FIFO "vintage pot": each year's income enters the pot as a new vintage dated to that year; each
  year's expense and *net positive* reserve allocation consume the pot oldest-vintage-first (a
  reserve allocation legitimately removes funds from the §55 clock); a *net negative* allocation
  (dissolution) returns funds to the pot as a **new, current-year vintage** -- a deliberate,
  explicitly flagged simplification (the alternative -- re-deriving the dissolved funds' original
  vintage -- is genuinely ambiguous, and re-dating is the conservative choice, never less likely to
  flag work as time-barred). A vintage is overdue once strictly more than `timelyUseYears`
  (`UseOfFundsStatementDto.timelyUseYears`, currently `2` -- the historically-interpreted "by the
  end of the second fiscal year following receipt", surfaced as data rather than a silent constant
  because it should be verified against current AO guidance) fiscal years have elapsed since its
  vintage year. The pot is floored at `0`, never negative.

**The clock is anchored at inception and is org-wide, not per-sphere.** `statement` always rolls the
FIFO pot forward from the organization's *earliest* fiscal year with any activity through
`toFiscalYear` -- never merely from `fromFiscalYear` -- because windowing the roll-forward itself
would silently corrupt the reported Mittelvortrag/overdue figures whenever the requested window
starts after inception; `fromFiscalYear`/`toFiscalYear` only slice which years are *reported*.
The clock is deliberately aggregate (not split per `GemeinnuetzigkeitSphere`) because surpluses
legitimately move between spheres and a per-sphere clock would misstate the obligation --
per-sphere received/used is informational only, same "cash-line sphere is informational" tradeoff
V0.3.3 already accepted for Bestandskonten. `getUseOfFundsStatement` (role: TREASURER/BOARD/ADMIN,
same as every other statement method) validates `fromFiscalYear <= toFiscalYear` and both years in
range with `BadRequestException`, then loads one `UseOfFundsCalculator.YearFacts` per calendar year
with any `POSTED` activity (`AccountingService.loadYearFacts`) and delegates to the pure calculator.
Only `POSTED` postings contribute -- same "DRAFT is provisional" rule as every other statement.

**Reserve movements are keyed by the ledger account's `reserveType`, never by `posting.sphere`.**
A reserve-transfer posting's two lines are `EQUITY` and still each carry the mandatory (V0.3.3)
per-line sphere, which remains purely informational for these lines -- the reserve-movement loader
groups strictly by the target account's `reserveType`, so a transfer whose debit and credit lines
happen to name different spheres still books cleanly and is counted exactly once under its reserve
type.

Explicitly, deliberately out of scope for V0.3.4 (see `UseOfFundsCalculator`/`ReserveType` KDoc for
the full list, each flagged individually): the §62 Abs.1 Nr.3 freie-Rücklage percentage-cap
enforcement; automatic application of the §55 Abs.1 Nr.5 Satz 4 small-organization
(≤45.000 EUR) exemption; any automated "Gemeinnützigkeit lost/retained" verdict or certification;
§64 AO Freigrenze evaluation (later compliance-bundle territory); Kassenbuch (V0.3.5); general-purpose
Kostenstellen (V0.3.6); non-calendar fiscal years; and any client/KVision UI (this is a backend-only
wave).
